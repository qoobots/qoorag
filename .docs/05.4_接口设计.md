## 1. 概述与范围

### 1.1 目的

将应用架构（04 §4）的接口契约骨架落地为**可执行的接口规范**：统一鉴权、统一响应、统一错误码，给出每个接口的方法/路径/请求体/响应/错误，并产出 OpenAPI 3.0.3 文件供 Swagger/Redoc/代码生成使用。

### 1.2 范围

- **覆盖**：通用约定、错误码规范、全部 18 个接口清单、关键调用示例、OpenAPI 使用方式。
- **不覆盖**：逐字段 schema（见 `openapi.yaml`）、数据库字段（见 03 §4）、类级协作（见 05.3）。

### 1.3 接口总览

| 分组 | 数量 | 鉴权 | 路径前缀 |
| --- | --- | --- | --- |
| 认证 Auth | 3 | 登录免鉴权；其余会话令牌 | `/api/auth` |
| 系统管理 SystemAdmin | 6 | 会话令牌 + 系统管理员角色 | `/api/admin` |
| 知识库管理 KnowledgeBase | 9 | 会话令牌 + 知识管理员角色 | `/api/kb` |
| 对外 API ExternalAPI | 2 | API Key（Bearer） | `/api/v1` |
| **合计** | **20** | — | — |

> 注：系统管理 6 = `users`(GET/POST) + `users/{id}/roles`(POST) + `users/{id}/status`(PUT) + `roles`(GET/POST)；知识库管理 9 = `kb`(GET/POST) + `kb/{id}`(DELETE) + `permissions`(GET/POST) + `permissions/{permId}`(DELETE) + `apikeys`(GET/POST) + `apikeys/{keyId}`(DELETE)。

## 2. 通用约定

### 2.1 基础信息

- **Base URL（开发）**：`http://localhost:8080`（Spring Boot 3.2.5，端口 8080，无 context-path；见 `application.yml` / `pom.xml`）。
- **Base URL（生产）**：`https://<host>`（前置反向代理/Ingress，**全链路 TLS**，见 05 §8.2）。
- **内容类型**：请求/响应均为 `application/json;charset=UTF-8`；文件上传上限 50MB（`spring.servlet.multipart`，见 `application.yml`）。
- **时间格式**：ISO-8601 `date-time`（如 `2026-07-21T10:30:00`）。

### 2.2 鉴权（两种 Bearer）

| 场景 | Scheme | 请求头 | 说明 |
| --- | --- | --- | --- |
| 管理接口 | 会话令牌 | `Authorization: Bearer <sessionToken>` | 由 `POST /api/auth/login` 签发；内存态（`AuthService.sessions`），生产改 Redis（05 §6/§8） |
| 对外接口 | API Key | `Authorization: Bearer <apiKey>` | 按知识库签发（SHA-256 存储）；自动继承绑定 KB 的 `tenant_id` 隔离与 RBAC 检索权限（05 §8.1） |

- 白名单：仅 `POST /api/auth/login` 免鉴权；其余 `/api/**` 经 `AuthInterceptor`（05 §8.1 / 05.3 §3.2）。
- 路径区分：`/api/v1/**` 走 API Key 校验，其余走会话令牌校验（05.3 §2.3）。

### 2.3 统一响应结构

所有接口返回统一包装（对应 `common.Result`）：

```json
{ "code": 0, "message": "ok", "data": <业务对象或数组或 null> }
```

- `code = 0` 表示成功；非 0 为业务/错误码（见 §3）。
- `data` 类型随接口变化（对象/数组/null），详见 `openapi.yaml` 各 `Resp*` 响应。
- 鉴权失败（401）由 `AuthInterceptor` 直接写 `{"code":401,"message":"..."}`（不经过 `Result` 包装，但结构兼容），HTTP 状态 401。

### 2.4 分页与排序

MVP 接口均为全量返回（无游标/页码参数）；大规模列表的分页（offset/limit、游标）与排序为后续演进（见 04 §6 / 05 §11）。

## 3. 错误码规范

统一错误码（`code` 字段），HTTP 状态码与业务码对应如下：

| 业务码 | HTTP | 含义 | 触发场景 |
| --- | --- | --- | --- |
| `0` | 200 | 成功 | — |
| `40001` | 400 | 参数校验失败 | 请求体缺必填项/类型错误（`@Validated`，待接入） |
| `40101` | 401 | 未登录或令牌缺失 | 管理接口缺/错 `Authorization: Bearer` |
| `40102` | 401 | 令牌无效或已过期 | `sessions` 中无该 token |
| `40103` | 401 | 缺少 API Key | `/api/v1/**` 缺 `Authorization: Bearer` |
| `40104` | 401 | API Key 无效或已吊销 | `key_hash` 不匹配或 `status≠ACTIVE`/`deleted_at≠null` |
| `40301` | 403 | 无权限：需要系统管理员角色 | 非系统管理员访问 `/api/admin` |
| `40302` | 403 | 无权限：需要知识管理员角色 | 非知识管理员访问 `/api/kb` |
| `40303` | 403 | 该 Key 无此知识库检索权限 | API Key 绑定的 KB 与请求 KB 不匹配/未授权 |
| `40401` | 404 | 资源不存在 | KB/用户/角色 ID 不存在或已软删 |
| `40901` | 409 | 用户名已存在 | `users.username` 唯一约束冲突 |
| `40902` | 409 | 角色名已存在 | `roles.name` 唯一约束冲突 |
| `42201` | 422 | 账号已被停用 | `status≠ACTIVE` 或 `deleted_at≠null` |
| `42901` | 429 | 超出 Key 速率限制 | 超过 `api_key.rate_limit`（按 Key 维度，待实现） |
| `50001` | 500 | 服务器内部错误 | 未预期异常（建议接 `@ControllerAdvice` 统一包装，见 05.3 §4.2） |

> 错误响应体：`{"code": <业务码>, "message": "<中文描述>"}`（`data` 省略）。`401` 由拦截器直接写出，其余由业务/异常处理写出。

## 4. 接口清单

### 4.1 认证（Auth）

| 方法 | 路径 | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/auth/login` | 免 | 登录，返回 `token`/`userId`/`roles` |
| POST | `/api/auth/logout` | 会话 | 登出（销毁令牌） |
| GET | `/api/auth/me` | 会话 | 当前会话 `userId`/`tenantId`/`roles` |

### 4.2 系统管理（SystemAdmin，需系统管理员角色）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/admin/users` | 列出当前租户未删除用户（写审计 `LIST_USERS`） |
| POST | `/api/admin/users` | 新增用户 `{username,password,displayName,roleIds}`（写审计 `CREATE_USER`） |
| POST | `/api/admin/users/{id}/roles` | 分配/替换角色 `{roleIds}`（写审计 `ASSIGN_ROLES`） |
| PUT | `/api/admin/users/{id}/status` | 启用/停用 `{status: ACTIVE\|DISABLED}`（写审计 `SET_USER_STATUS`） |
| GET | `/api/admin/roles` | 列出角色 |
| POST | `/api/admin/roles` | 新增角色 `{name,description}`（写审计 `CREATE_ROLE`） |

### 4.3 知识库管理（KnowledgeBase，需知识管理员角色）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/kb` | 列出当前租户知识库 |
| POST | `/api/kb` | 创建 `{name,description,dataClassification}`（写审计 `CREATE_KB`） |
| DELETE | `/api/kb/{id}` | 软删除（写审计 `DELETE_KB`，进入保留期，见 05 §9.4） |
| GET | `/api/kb/{id}/permissions` | 列出检索权限 |
| POST | `/api/kb/{id}/permissions` | 授权 `{targetType: ROLE\|USER\|EXTERNAL, targetId}`（写审计 `GRANT_KB_PERMISSION`） |
| DELETE | `/api/kb/{id}/permissions/{permId}` | 收回权限（写审计 `REVOKE_KB_PERMISSION`） |
| GET | `/api/kb/{id}/apikeys` | 列出 API Key |
| POST | `/api/kb/{id}/apikeys` | 签发 Key（明文仅一次返回，写审计 `CREATE_API_KEY`） |
| DELETE | `/api/kb/{id}/apikeys/{keyId}` | 吊销 Key（写审计 `REVOKE_API_KEY`） |

### 4.4 对外 API（ExternalAPI，API Key 鉴权，OpenAI 兼容）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/v1/retrieve` | 检索召回片段 `{query, top_k?, kb_id?}` → `{query, top_k, chunks[]}`（骨架 TODO） |
| POST | `/api/v1/chat` | 检索+生成问答 `{query, top_k?, model?, stream?, kb_id?}` → `{answer, sources[], model, usage}`（骨架 TODO，已落 `qa_trace`） |

> `/api/v1/*` 为骨架：真实 Embedding+pgvector 检索与 LLM 生成待接入（05 §9.2 / §9.5 / 05.3 §3.5）。

## 5. 关键调用示例

### 5.1 登录 → 管理接口（会话令牌）

```bash
# 1) 登录
curl -X POST http://localhost:8080/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"123456"}'
# => {"code":0,"message":"ok","data":{"token":"c1f0e2...","userId":1,"roles":["系统管理员","知识管理员"]}}

# 2) 携带令牌调用管理接口
curl http://localhost:8080/api/kb \
  -H 'Authorization: Bearer c1f0e2...'
# => {"code":0,"message":"ok","data":[{"id":1,"name":"产品手册库",...}]}
```

### 5.2 对外检索/问答（API Key）

```bash
# 检索（骨架：chunks 当前为空，待接入向量库）
curl -X POST http://localhost:8080/api/v1/retrieve \
  -H 'Authorization: Bearer qk_9f2a...c1b2e3d4' \
  -H 'Content-Type: application/json' \
  -d '{"query":"如何重置密码？","top_k":5}'
# => {"code":0,"message":"ok","data":{"query":"如何重置密码？","top_k":5,"chunks":[]}}

# 问答（骨架：answer 为占位，已落 qa_trace）
curl -X POST http://localhost:8080/api/v1/chat \
  -H 'Authorization: Bearer qk_9f2a...c1b2e3d4' \
  -H 'Content-Type: application/json' \
  -d '{"query":"请总结退货政策要点","top_k":5}'
# => {"code":0,"message":"ok","data":{"answer":"[骨架占位]...","sources":[]}}
```

### 5.3 错误示例

```bash
curl http://localhost:8080/api/kb        # 缺令牌
# => HTTP 401, {"code":401,"message":"未登录或令牌缺失"}

curl -X POST http://localhost:8080/api/admin/users \
  -H 'Authorization: Bearer <普通用户令牌>' \
  -d '{"username":"bob","password":"x"}'
# => HTTP 403, {"code":40301,"message":"无权限：需要系统管理员角色"}
```

## 6. OpenAPI 规范使用

- **规范文件**：`.docs/openapi.yaml`（OpenAPI 3.0.3，含 `sessionToken` / `apiKey` 两种 securityScheme、全部 schema 与响应、错误码示例；`/api/v1/*` 标注 `x-status: skeleton`）。
- **本地预览**（任选其一）：
  - Swagger UI：`npx @redocly/cli serve .docs/openapi.yaml` 或导入 Swagger Editor。
  - Redoc：`npx redoc-cli serve .docs/openapi.yaml`。
- **客户端/服务端代码生成**：使用 OpenAPI Generator（`openapi-generator-cli`）按 `openapi.yaml` 生成 Java/TS SDK 或 Spring 控制器骨架。
- **契约校验**：CI 中可加 `redocly lint .docs/openapi.yaml` 保证规范合法。
