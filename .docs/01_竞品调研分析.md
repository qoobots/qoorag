## 2. 逐维度深度对比

### 2.1 产品定位与差异化
- **Dify**：最"宽"，是 LLM 应用开发平台，RAG 只是其中一环；适合要工作流/Agent 编排的团队。
- **RAGFlow**：最"深"，专注 RAG 与深度文档理解（复杂 PDF/表格解析强），定位"为 LLM 构建更优上下文层"。
- **FastGPT**：知识库 + 工作流 + Agent 一体，强调企业级权限与私有化，商业化成熟。
- **MaxKB**：聚焦企业知识库问答，轻量、易部署（1Panel 生态）。
- **AnythingLLM**：最轻量，MIT、桌面友好，适合个人/小团队快速起私有知识库。
- **qoorag 切入点**：偏"平台/产品 + 多用户多知识库 + 每库独立六层 RAG 链路 + 企业合规"。与 RAGFlow 同属 RAG 纵深，但强调**每个知识库独立可插拔链路**与**企业级合规治理**，与 Dify/FastGPT 的"应用/智能体平台"定位形成差异。

### 2.2 开源协议与商业化
- Dify / RAGFlow / FastGPT 均为 Apache 2.0 系（FastGPT、Dify 有附加条款）。AnythingLLM 为最宽松的 MIT。
- 观察：**主流竞品均选 Apache 2.0 / MIT 这类企业友好协议**，qoorag 已定 **Apache 2.0**，与之对齐，利于企业采用与二次开发。
- 商业化普遍采用"开源核心 + 商业/企业版（SSO、审计、多租户增强、计费）"分层。qoorag 已决策**不内置配额/计费**（不做也不考虑），差异化聚焦在"每库独立链路 + 企业合规治理"，与开源核心定位一致。

### 2.3 多用户 / 多租户与用户模型
- 普遍支持多用户，但**企业场景下均为"管理员创建 / SSO 同步"，不开放公网自助注册**。
- Dify 用 Workspace=Tenant 做租户隔离；FastGPT 用 SSO+RBAC；AnythingLLM 用多用户实例 + 权限。
- **对 qoorag 的启示（待定项 2）**：采用"管理员创建用户 / SSO 同步自动开通"，不开放公网注册，符合企业级惯例，也与 4.3/4.4 一致。

### 2.4 知识库共享模型
- 知识库普遍归属某个 Workspace/团队，**在授权范围内复用与共享**，跨租户默认隔离。
- AnythingLLM 的"工作区"模型最清晰：工作区内共享文档，工作区间隔离。
- **对 qoorag 的启示（待定项 3）**：当前"用户私有、严格隔离"可保留为默认；建议增加**基于 RBAC 的共享**（如将某知识库只读共享给指定用户/角色），以对齐企业"部门知识库"诉求，且不破坏隔离底线。

### 2.5 配额 / 计费
- **关键发现：主流开源 RAG 平台几乎都不内置按用户/按调用的配额计费**（计费多在商业云版）。
- **对 qoorag 的启示（原待定项 5/6）**：qoorag 曾规划"按用户积分 + 固定向量库容量 + 超额申请"作为差异化能力。**❌ 决策更新（2026-07-21）：配额/计费/积分折算体系已决策彻底移除——不做也不考虑，含产品化/云化后续演进**；差异化聚焦在"每库独立可插拔链路 + 企业级合规治理（审计/留痕/数据分级/不出域/RLS 隔离）"。

### 2.6 外部资源与数据驻留（重要张力）
- 所有竞品都支持配置**外部 LLM API**（OpenAI、通义、DeepSeek 等）与多种向量库；但同时都强调**私有化部署 + 本地模型（Ollama）**以保数据不出域。
- 外部模型意味着知识片段会发往企业外，与"数据不出域"天然冲突——竞品通过"私有化 + 本地模型"选项来化解，而非禁止外部资源。
- **对 qoorag 的启示（待定项 4）**：保留 4.7 第 3 层"用户自带外部资源"，但**默认仅限企业内网部署的实例**；若允许用户配公网外部模型，需显式风险提示 + 审批，并在 4.8 合规中标注"外部资源 = 数据出域例外"。

### 2.7 多租户隔离实现
- Dify 等采用 **tenant_id 行级隔离**（同一库、每行带租户标识），企业版可加重隔离。
- **对 qoorag 的启示（待定项 7）**：单 PostgreSQL 库下，采用**行级隔离（user_id / kb_id 字段）+ PostgreSQL 行级安全策略（RLS）**最为务实，部署简单且满足等保"越权访问防护"要求；不必分库/分 schema，降低运维复杂度。

### 2.8 API 鉴权与限流
- 普遍提供 **API Key**；Dify 按应用、FastGPT 按 OpenAI 兼容接口、AnythingLLM 提供开发者 API。
- **对 qoorag 的启示（待定项 8）**：提供 **OpenAI 兼容 API**，鉴权用 **API Key（建议按知识库粒度签发，可归属到用户）**；限流按 API Key 维度速率限制（非配额计费），MVP 即做基础限流。

### 2.9 知识库生命周期
- 竞品普遍：删除知识库即清理其向量与文档；审计/操作日志通常独立留存。
- **对 qoorag 的启示（待定项 9）**：删除知识库清理向量数据，但**审计日志独立保留**（满足 4.8"防篡改留存、日志留存≥6 个月"），删除操作本身也记入审计。

## 3. 对 qoorag 9 项待定决策的启示汇总

| # | 待定项 | 竞品参照结论 | 建议 |
| --- | --- | --- | --- |
| 1 | MVP 范围 vs 已定架构 | 竞品 MVP 均含核心 RAG + 基础权限，计费后置 | MVP = 基础审计 + 数据分级字段（配额拦截已决策移除）；审批流/密评/脱敏后置 |

| 5 | 默认额度数值 | 竞品无内置，无标准值可抄 | ❌ 已决策彻底移除（不做也不考虑） |

| 8 | API 鉴权与限流 | API Key 为主，OpenAI 兼容 | OpenAI 兼容 API + 按知识库签 Key + 按 Key 维度限流（非配额） |
| 9 | 知识库生命周期 | 删库清向量、审计独立留存 | 删库清向量；审计日志独立保留≥6 个月 |

## 4. 结论：qoorag 的差异化切入点

1. **每知识库独立可插拔六层链路 + 配置随库走**：比 Dify/FastGPT 的"应用级"更细，比 RAGFlow 的"引擎级"更产品化。
2. ~~企业级配额/计费（按用户积分 + 向量库容量）作为开源核心能力~~（❌ 已决策彻底移除：不做也不考虑，含产品化/云化后续）。
3. **合规与等保内建（审计/留痕/数据分级/不出域）**：对齐企业采购门槛，与私有化部署卖点一致。
4. **Java Spring Boot + 纯前端 + PostgreSQL/pgvector 统一栈**：区别于主流 Python/Node 实现，契合企业 Java 技术栈与运维习惯（需权衡：RAG 生态 Python 工具更多，抽象层要做好）。
5. **独立微服务 + 可接入 IDaaS**：与"企业智能化"工程其他子系统松耦合，定位清晰。
