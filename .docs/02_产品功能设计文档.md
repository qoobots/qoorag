## 2. 产品概述

### 2.1 一句话定位

**qoorag** 是一个企业级、可私有化部署、模块化可扩展的开源 RAG（检索增强生成）平台：支持多用户管理，每位用户可管理多个相互隔离的知识库，每个知识库拥有独立且可插拔的完整 RAG 链路，让企业以可控、合规、低成本的方式把内部私有知识接入大模型。

### 2.2 产品形态

- **平台/产品形态**：多用户 × 多知识库，用户间数据隔离，知识库间链路配置独立。
- **独立微服务**：作为"企业智能化"工程中自治子系统，通过标准 API 对外提供检索/问答能力，身份可接入企业 IDaaS，与统一门户/智能体平台/数据中台松耦合。
- **私有化优先**：全栈可部署于企业内网，知识数据不出域。

## 3. 目标用户与角色（RBAC）

### 3.1 用户画像

| 用户类型 | 核心诉求 | 主要使用页面 |
| --- | --- | --- |
| 企业 IT / 平台管理员（系统管理员） | 统一部署、身份对接、安全合规、用户治理 | 系统管理页面 |
| 业务部门 / 知识库 owner（知识管理员） | 自助搭建本部门知识库问答 | RAG 管理页面 |
| 普通员工（内部检索者） | 自然语言获取知识 | 对话/检索界面（被授权后） |
| AI 应用开发者 | 将检索/问答集成到业务系统 | 对外 API（API Key） |
| 外部协作方（经认证） | 有限范围内检索授权知识库 | 对外 API / 受限界面 |

### 3.2 角色与权限矩阵

采用"用户—角色"RBAC 雏形，初始写入两角色；MVP 可单角色，后续可扩展多角色与细粒度菜单权限。

| 角色 | 系统管理页 | RAG 管理页 | 检索/问答 | 对外 API | 审计查看 |
| --- | --- | --- | --- | --- | --- |
| 系统管理员 | ✅ 全权 | ❌（除非兼岗） | 视授权 | 视授权 | ✅（全平台） |
| 知识管理员 | ❌ | ✅ 本租户知识库 | 视授权 | ✅（本库 Key） | 本库范围 |

**权限继承原则**：

- 知识库**默认私有**，仅创建者/同知识管理员角色可见可用。
- 检索/问答为只读使用权限，由知识管理员显式授权；**管理权不随检索授权转移**。
- 审计日志仅系统管理员/安全合规角色可见，不随知识库删除而删除。

### 3.3 默认初始数据

- 默认用户：`admin` / `123456`（首次登录强制改密）。
- 默认角色：`系统管理员`、`知识管理员`，写入角色表；`admin` 绑定两角色。

### 3.4 用户开通方式（企业内部，不开放公网注册）

1. **系统管理员创建**（MVP 默认）：在系统管理页新增用户并分配角色。
2. **企业身份同步自动开通**（可插拔，后置）：接入 SSO/OAuth2/LDAP(AD)/企业微信/钉钉/飞书后，首次登录按策略自动建号并映射角色。接入优先级：SSO/OAuth2 优先，其余依次扩展。

## 5. 功能模块详细设计

### 5.1 系统管理（系统管理员）

**5.1.1 用户管理**

- 新增用户：账号、姓名、初始密码（强制首次改密）、归属租户(tenant_id)、状态（启用/停用）。
- 编辑/停用：停用后禁止登录，历史审计与问答留痕保留。
- 不开放自助注册；无公网注册入口。

**5.1.2 角色管理**

- 新增/编辑角色，定义可见菜单与权限点（MVP 先用两默认角色，支持扩展）。
- 用户-角色分配/收回。

**5.1.3 资源池管理（核心：集中纳管，禁止外部自带）**

- 系统管理员统一接入并维护：
  - LLM（生成）端点与密钥；
  - Embedding 模型端点与密钥；
  - Rerank 模型（可选）端点；
  - 向量库实例（pgvector 等）。
- 知识管理员**只能从资源池选择**，不得自行接入外部模型/外部向量库（架构上保障数据不出域，见 4.7、4.8）。
- 资源池对全体知识管理员可见可选；MVP 资源池条目来自配置文件/初始化，后续可界面化管理。

**5.1.4 审计日志查看**

- 查看全平台操作审计（登录、权限变更、知识库增删改、配置变更、Key 操作等）。
- 只读、不可删除/篡改，支持按时间/操作人/对象筛选。

### 5.2 知识库管理（知识管理员）

**5.2.1 知识库创建**

- 必填：名称、描述、归属租户(tenant_id)、数据分级（公开/内部/机密/绝密，可配置）。
- 六层配置：从资源池选择 数据接入策略 / 分块策略 / Embedding / 向量库 / 检索方式 / Rerank / LLM / Prompt 模板（MVP 提供一套默认配置，可后续调）。
- 创建后默认**私有**。

**5.2.2 文档管理（数据接入→处理→索引）**

- 上传格式（MVP）：PDF、Markdown、TXT（后续扩展 Word/PPT/Excel/HTML、数据源连接器）。
- 处理流水线：解析 → 分块（固定/语义/标题层级，MVP 固定长度+可配） → Embedding 向量化 → 写入向量库（带 tenant_id、kb_id）。
- 状态可见：上传中/解析中/向量化中/已入库/失败（失败可重试）。
- 文档级元数据：来源、上传时间、所属知识库、数据分级。

**5.2.3 检索与问答（知识库内）**

- 用户输入问题 → 向量检索（+关键词 BM25，MVP 可先纯向量） → （可选 Rerank） → 召回 Top-K 片段 → LLM 生成答案并标注来源引用。
- 多轮对话上下文；流式输出；无答案时明确告知（幻觉抑制）。

**5.2.4 检索权限管理（RBAC 共享）**

- 授权对象：内部角色 / 部门 / 具体用户；经认证的外部用户（外部凭证）。
- 权限粒度：检索/问答（只读）。管理权不随授权转移。
- 默认私有；授权/收回全程留痕。
- 与数据分级联动：高密级库仅授权且具相应密级权限者可检索。

**5.2.5 删除与生命周期（软删+保留期）**

- 删除进入软删除（标记 deleted_at，暂停检索）。
- 保留期内可恢复；过期由定时任务物理清理**业务数据**（文档/分块/向量）。
- **审计日志与问答留痕独立保留**，不随删库删除（等保取证）。
- 清理受 tenant_id 约束，不影响其他租户/库。

### 5.3 资源池（系统管理员纳管，详见 5.1.3）

- 两层资源模型：平台资源池（复用+安全合规） + 知识库独立配置（隔离+灵活）。
- MVP 资源池通过配置/初始化提供；知识库在池内自由编排、互不干扰。

### 5.4 检索与问答（授权用户视角）

- 见 5.2.3。普通员工/外部用户仅能在被授权知识库内检索/问答。

### 5.5 对外 API（OpenAI 兼容）

**5.5.1 鉴权**

- 按知识库签发 API Key；调用凭 Key 访问**对应知识库**（继承检索权限与 tenant_id 隔离）。
- Key 仅展示一次、加密存储；支持轮换/吊销；传输 TLS。

**5.5.2 接口（MVP 骨架）**

- `POST /api/v1/retrieve`：检索召回片段。
- `POST /api/v1/chat`：检索+生成问答。
- 风格对齐 OpenAI 兼容（后续完善请求/响应契约，见 03 接口设计）。

**5.5.3 限流**

- 按 Key 维度速率限制（每分钟/每小时请求数与 Token 上限），非配额计费；策略由系统管理员配置。
- 每次调用记录 Key、知识库、调用方、时间、用量，纳入审计与问答留痕。

### 5.6 合规与可观测（贯穿全平台）

**5.6.1 审计（Audit）**

- 操作审计：登录、权限变更、知识库增删改、配置变更、Key 操作等，记录操作人/时间/对象/前后值。
- 全链路可追溯：检索与生成过程（模型、召回片段、Prompt、输出、参数）可回溯。
- 防篡改：只读、不可删、定期归档。

**5.6.2 留痕（Traceability）**

- 问答留痕（QaTrace）：问题、召回来源、答案、模型与参数、时间戳、用户身份。
- 知识库操作留痕：上传/解析/向量化/配置变更全程留痕。
- 留存周期：满足等保（一般≥6 个月），到期归档/安全销毁。

**5.6.3 数据分级**

- 知识库/文档标注敏感级别；分级访问控制；PII 脱敏（MVP 仅保留字段，脱敏/密评后置）。

## 6. 关键业务流程（用例）

### 6.1 系统初始化与开通

```
[系统管理员] 首次登录 admin → 强制改密
    → 创建用户（如 知识管理员 A）并分配角色
    → 维护资源池（LLM/Embedding/向量库）
```

### 6.2 知识库搭建到问答闭环

```
[知识管理员] 登录 RAG 管理页
    → 创建知识库（选默认六层配置，默认私有）
    → 上传文档（PDF/MD/TXT）
    → 系统自动：解析→分块→向量化→入库
    → 授权检索对象（角色/部门/用户/外部）
    → 在库内问答（检索+生成+来源引用）
```

### 6.3 对外 API 调用

```
[知识管理员] 签发 API Key → 配置限流
[开发者] 持 Key 调用 /api/v1/chat
    → 拦截器校验 Key→定位知识库→校验 tenant_id 隔离与检索权限
    → 检索+生成 → 返回答案（带来源）
    → 调用记录写入审计与 QaTrace
```

### 6.4 知识库删除与合规留存

```
[知识管理员] 删除知识库 → 软删除（暂停检索）
    → 保留期内可恢复
    → 过期定时任务物理清理业务数据（文档/分块/向量）
    → 审计日志 + 问答留痕 独立保留（不删）
```

## 7. 非功能性需求

### 7.1 隔离与安全

- **多租户隔离**：单 PostgreSQL 库 + `tenant_id` 行级标记 + PostgreSQL RLS 双重隔离；应用层强制带 tenant_id，数据库层 RLS 兜底防越权。
- **数据不出域**：全私有化部署；模型/向量库统一系统管理员纳管，知识管理员不得接外部资源。
- **传输/存储加密**：TLS 传输；敏感字段/向量库加密（密钥由企业 KMS，MVP 预留接口）。
- **身份鉴别**：对接 SSO/LDAP（可插拔）；弱口令策略、登录失败锁定、MFA 可选（后置）。

### 7.2 性能（MVP 目标）

- 单知识库问答 P95 < 5s（含检索+生成，受模型时延影响）。
- 文档入库吞吐：单文档（<20MB）解析+向量化 < 60s（受 Embedding 服务影响）。
- 支持按 Key 限流，防止单 Key 滥用。

### 7.3 可用性/可运维

- Docker Compose 一键启动（Spring Boot + 前端静态 + PostgreSQL(pgvector) + Redis）。
- 操作日志、调用日志可查；平台级监控（活跃用户、调用量）后置。

### 7.4 合规（等保 2.0 对齐）

- MVP 落地：基础操作审计、问答留痕、数据分级字段、tenant_id 隔离。
- 后置：脱敏、密评、内容安全过滤细化、≥6 个月留存策略固化。

### 7.5 兼容性

- 后端 Java 17 + Spring Boot 3.2；前端纯 HTML5/CSS3/JS；数据库 PostgreSQL + pgvector。
- 模型服务通过 HTTP 调用（兼容 OpenAI 等 API 规范），AI 能力以外部服务集成。

## 8. MVP 范围界定

### 8.1 In Scope（第一个可用版本）

1. 内置 `admin` 登录系统管理页；系统管理员创建用户、角色并分配（如知识管理员）。
2. 知识管理员登录 RAG 管理页，创建知识库（命名 + 选默认六层配置）。
3. 知识库内上传文档（PDF/Markdown/TXT）→ 自动解析 → 分块 → 向量化 → 入库。
4. 知识库内问答（向量检索 + LLM 生成 + 来源引用）。
5. 检索权限 RBAC 授权（角色/部门/用户/外部），默认私有。
6. 按知识库签发 API Key + 基础按 Key 限流 + 调用审计；`/api/v1/retrieve`、`/api/v1/chat` 骨架。
7. 软删除 + 保留期 + 业务数据清理 + 审计/留痕独立留存。
8. 基础操作审计日志、问答留痕、数据分级字段、tenant_id 隔离（+基础 RLS）。
9. 私有化 Docker Compose 一键启动。

### 8.2 Out of Scope（MVP 不做）

- 配额/计费拦截（固定积分+容量+耗尽拦截，属产品化/云化能力，企业内部 MVP 不实现；仅保留字段）。
- 多数据源连接器（Notion/Confluence/DB/URL 抓取等）。
- 复杂权限/角色体系、细粒度菜单权限。
- 集群、评估平台、检索质量评估工具。
- 脱敏、密评等高级合规项（保留字段与扩展位）。
- 外部模型/外部向量库接入（统一系统管理员纳管，知识管理员不得自带）。
- 企业身份对接 SSO/LDAP/IM 的正式接入（保留可插拔扩展位，MVP 用本地账户）。

## 9. 数据实体概要（详见 03 系统设计）

核心实体（均带 `tenant_id` 与 `deleted_at`）：

| 实体 | 说明 |
| --- | --- |
| Tenant | 租户/组织单元 |
| User | 用户账号 |
| Role | 角色 |
| KnowledgeBase | 知识库（含六层配置、数据分级） |
| KbPermission | 知识库检索权限授权记录 |
| Document | 文档元信息 |
| Chunk | 文档分块 |
| VectorData | 向量数据（embedding，带 tenant_id/kb_id） |
| ApiKey | 知识库 API Key（加密存储、限流配置） |
| AuditLog | 操作审计日志（独立留存） |
| QaTrace | 问答留痕（独立留存） |

关系要点：User—Role 多对多；KnowledgeBase 归属 Tenant；KbPermission 关联 KB 与授权对象；Document→Chunk→VectorData 级联；ApiKey 绑定 KB；AuditLog/QaTrace 独立、不随删库删除。

## 10. 验收标准（MVP）

| 编号 | 验收项 | 通过标准 |
| --- | --- | --- |
| AC-01 | 登录与角色 | admin 登录强制改密；系统管理员可创建用户并分配角色 |
| AC-02 | 知识库创建 | 知识管理员可创建知识库并选择默认六层配置，默认私有 |
| AC-03 | 文档入库 | 上传 PDF/MD/TXT 后自动解析→分块→向量化→入库，状态可见 |
| AC-04 | 问答闭环 | 库内问答返回答案且标注来源引用，支持多轮与流式 |
| AC-05 | 检索授权 | 可按角色/部门/用户/外部授权检索；未授权不可检索 |
| AC-06 | 对外 API | 持有效 Key 可调用 /retrieve、/chat；无效/吊销 Key 被拒 |
| AC-07 | 限流 | 单 Key 超速率限制被限流 |
| AC-08 | 生命周期 | 删除进入软删；保留期内可恢复；过期业务数据清理、审计/留痕保留 |
| AC-09 | 隔离 | 跨 tenant_id 数据不可互查（应用层+RLS 兜底） |
| AC-10 | 审计留痕 | 关键操作与问答写入审计/QaTrace，只读不可删 |
| AC-11 | 部署 | Docker Compose 一键启动，内网可访问 |
