## 1. 一句话愿景

打造一个**偏平台 / 产品形态、可自托管、模块化可扩展**的开源 RAG（检索增强生成）系统：支持多用户管理，每位用户可管理多个相互隔离的知识库，每个知识库都拥有独立且可插拔的完整 RAG 链路，让个人和企业都能低成本地把自己的私有知识接入大模型。

---

## 2. 我们想解决什么问题

- 大模型缺乏私有 / 实时 / 领域知识，容易"一本正经地胡说"。
- 现有 RAG 方案要么太重（企业级、难部署），要么太轻（demo 级、不可用于生产）。
- 用户搭建 RAG 需要拼接：文档解析、切分、向量化、检索、重排、生成……链路长、门槛高。
- 缺乏一个"约定优于配置"、能快速跑通全流程、又能逐步深入定制的开源项目。

## 3. 目标用户画像

| 用户类型 | 核心诉求 | 使用场景 |
| --- | --- | --- |
| 个人开发者 / 极客 | 快速搭建私人知识库问答 | 本地文档、笔记、代码库问答 |
| 中小团队 | 内部知识库、客服助手 | 产品文档、FAQ、工单辅助 |
| 企业技术团队 | 可控、可私有化、可二次开发 | 内部知识管理、智能助手中台 |
| AI 应用开发者 | 作为底层能力集成到自己产品 | 提供检索 / 问答 API |

## 4. 产品形态与核心架构

### 4.1 总体形态：多用户 × 多知识库

项目定位为**平台 / 产品**形态，核心组织关系如下：

```
平台
└── 用户（账号）
    └── 知识库 A（独立、隔离）
    │   ├── 数据接入层
    │   ├── 处理与索引层
    │   ├── 检索层
    │   ├── 生成层
    │   ├── 应用与交互层
    │   └── 运维与可观测
    └── 知识库 B（独立、隔离）
        └── ……（同上六层，配置可完全不同）
```

- **多用户管理**：平台支持注册 / 登录的多个用户，用户之间数据隔离。
- **每个用户管理多个知识库**：用户可创建并管理多个知识库（如"产品文档库""内部 Wiki""客服 FAQ"），各自独立。
- **每个知识库拥有独立的六层能力**：知识库之间互不影响，且每个知识库可在六层上独立配置（例如 A 库用 BGE 向量 + pgvector，B 库用 OpenAI 向量 + Milvus；A 库用 DeepSeek 生成，B 库用本地 Ollama）。

### 4.2 知识库的独立六层（每个知识库可独立配置）

| 层级 | 知识库可独立配置的内容 |
| --- | --- |
| 数据接入层 | 接入的数据源、文档格式、同步策略、OCR 开关 |
| 处理与索引层 | 分块策略、Embedding 模型、向量库类型、元数据过滤 |
| 检索层 | 检索方式（向量 / 关键词 / 混合）、Rerank 模型、召回参数 |
| 生成层 | LLM 选型、Prompt 模板、对话策略、幻觉抑制 |
| 应用与交互层 | 对话 UI 配置、对外 API / 密钥、SDK 接入方式 |
| 运维与可观测 | 检索质量评估、调用日志、Token 与成本统计 |

> 关键设计原则：**配置随知识库走**。用户切换知识库即切换一整套 RAG 链路配置，平台负责隔离与编排。

### 4.3 平台级能力（跨用户 / 跨知识库）

- 用户与权限管理（注册、登录、角色、配额）。
- 全局模型 / 向量库资源池（供各知识库选用，统一接入与计费）。
- 租户级隔离（用户间数据、向量、日志互不越权）。
- 平台级监控与运营（活跃用户、调用量、成本大盘）。

## 5. 核心功能构想（发散）

### 5.1 数据接入层
- 多格式文档解析：PDF、Word、Markdown、HTML、TXT、PPT、Excel。
- 数据源连接器：本地文件、URL 抓取、Notion、Confluence、Git 仓库、数据库。
- 增量同步与定时更新。
- OCR 支持（扫描件、图片中的文字）。

### 5.2 处理与索引层
- 智能分块（Chunking）：固定长度、语义分块、按标题层级分块。
- 多种 Embedding 模型支持（OpenAI、BGE、M3E、本地模型等，可插拔）。
- 向量数据库可选：Milvus、Qdrant、pgvector、Chroma、Faiss。
- 元数据管理与过滤（来源、时间、权限标签）。

### 5.3 检索层
- 向量检索 + 关键词检索（BM25）混合检索。
- 重排（Rerank）模型接入。
- 多路召回、结果去重与合并。
- 引用溯源（答案标注来源片段）。

### 5.4 生成层
- 多 LLM 支持：OpenAI、Claude、通义、DeepSeek、本地 Ollama 等，统一抽象接口。
- Prompt 模板管理与可视化调试。
- 流式输出、多轮对话上下文。
- 幻觉抑制策略（无答案时明确告知）。

### 5.5 应用与交互层
- 开箱即用的 Web 管理后台（知识库管理、上传、调试）。
- 内置对话 UI。
- 完整的 RESTful / OpenAI 兼容 API。
- SDK（Python / JS）。

### 5.6 运维与可观测
- 检索质量评估工具（召回率、命中率）。
- 调用日志、Token 统计、成本分析。
- Docker / K8s 一键部署。

## 6. 差异化亮点（怎么和别人不一样）

- **真·开箱即用**：一条命令跑通，默认配置即可用。
- **全链路可插拔**：Embedding、向量库、LLM、Rerank 每一环都能替换。
- **本地优先 / 隐私优先**：支持全本地部署，数据不出内网。
- **轻量但可扩展**：从单机 demo 平滑升级到生产集群。
- **可观测的 RAG**：内置检索评估与调试能力，不做"黑盒魔法"。

## 7. 技术选型初步想法（待验证）

- 后端：Python（FastAPI）为主，兼顾 AI 生态。
- 前端：React / Vue + 现代 UI 组件库。
- 向量库：默认 Chroma / pgvector（轻量），可切换 Milvus / Qdrant。
- 任务队列：Celery / Redis（处理异步解析与索引）。
- 部署：Docker Compose 起步，后续 Helm Chart。

## 8. MVP 范围建议（第一个可用版本）

聚焦"最短可用闭环"，并体现平台 / 多用户 / 多知识库形态：

1. 用户注册 / 登录，进入个人工作台。
2. 用户创建知识库（命名 + 选择一套默认六层配置）。
3. 在知识库内上传文档（PDF / Markdown / TXT）→ 自动解析 → 分块 → 向量化 → 入库。
4. 在知识库内进行问答（向量检索 + LLM 生成 + 来源引用）。
5. 一个简单的 Web 界面完成"用户 → 知识库 → 上传 → 问答"全流程。
6. Docker Compose 一键启动。

**先不做**：多数据源连接器、复杂权限 / 角色体系、集群、评估平台、计费体系等（但需保留用户间数据隔离的基础结构）。

## 9. 潜在风险与挑战

- 文档解析质量参差（尤其是复杂 PDF / 表格）。
- 检索效果依赖 Embedding 与分块策略，需要大量调优。
- 多模型 / 多向量库抽象可能带来复杂度，需平衡灵活性与简洁性。
- 开源社区运营与文档质量决定项目能否被采用。
- 与 Dify、RAGFlow、FastGPT 等成熟项目的竞争与差异化。
