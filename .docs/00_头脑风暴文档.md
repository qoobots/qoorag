## 1. 一句话愿景

打造一个**企业级、偏平台 / 产品形态、可自托管、模块化可扩展**的开源 RAG（检索增强生成）系统：支持多用户管理，每位用户可管理多个相互隔离的知识库，每个知识库都拥有独立且可插拔的完整 RAG 链路，让企业能以可控、合规、低成本的方式把内部私有知识接入大模型，作为"企业智能化"工程的核心知识底座。

### 1.1 项目背景

- 本项目为企业级项目，是**企业内部"企业智能化"工程的一部分**，而非独立 ToC 产品。
- 目标是从"单点 AI 能力"走向"企业级知识智能化基础设施"：统一承载各业务线、各部门的私有知识问答与智能应用。
- 作为工程的一部分，需与既有企业 IT 体系对接（统一身份、组织架构、安全合规、运维监控），而非孤立存在。
- 强调**可控、可审计、可私有化部署**，满足企业数据安全与合规要求。

---

## 2. 我们想解决什么问题

- 大模型缺乏私有 / 实时 / 领域知识，容易"一本正经地胡说"。
- 企业内部知识分散在 Wiki、文档、工单、代码库、邮件中，员工查找与复用效率低，知识随人员流动流失。
- 现有 RAG 方案要么太重（企业级、难部署、绑定商业授权），要么太轻（demo 级、不可用于生产），缺少一个能接入企业 IT 体系、可私有化、可合规审计的开源平台。
- 各部门重复造轮子搭建问答机器人，缺乏统一的知识底座与治理能力。
- 搭建 RAG 需要拼接：文档解析、切分、向量化、检索、重排、生成……链路长、门槛高，企业难以独立运维。
- 缺乏一个"约定优于配置"、能快速跑通全流程、又能逐步深入定制、并满足企业安全合规要求的项目。

## 4. 产品形态与核心架构

### 4.1 总体形态：多用户 × 多知识库

项目定位为**平台 / 产品**形态，核心组织关系如下：

```
平台
└── 用户（账号）
    └── 知识库 A（独立、隔离）
    │   ├── 数据接入层
    │   ├── 处理与索引层
    │   ├── 检索层
    │   ├── 生成层
    │   ├── 应用与交互层
    │   └── 运维与可观测
    └── 知识库 B（独立、隔离）
        └── ……（同上六层，配置可完全不同）
```

- **多用户管理**：平台支持注册 / 登录的多个用户，用户之间数据隔离。
- **每个用户管理多个知识库**：用户可创建并管理多个知识库（如"产品文档库""内部 Wiki""客服 FAQ"），各自独立。
- **每个知识库拥有独立的六层能力**：知识库之间互不影响，且每个知识库可在六层上独立配置（例如 A 库用 BGE 向量 + pgvector，B 库用 OpenAI 向量 + Milvus；A 库用 DeepSeek 生成，B 库用本地 Ollama）。

### 4.2 知识库的独立六层（每个知识库可独立配置）

| 层级 | 知识库可独立配置的内容 |
| --- | --- |
| 数据接入层 | 接入的数据源、文档格式、同步策略、OCR 开关 |
| 处理与索引层 | 分块策略、Embedding 模型、向量库类型、元数据过滤 |
| 检索层 | 检索方式（向量 / 关键词 / 混合）、Rerank 模型、召回参数 |
| 生成层 | LLM 选型、Prompt 模板、对话策略、幻觉抑制 |
| 应用与交互层 | 对话 UI 配置、对外 API / 密钥、SDK 接入方式 |
| 运维与可观测 | 检索质量评估、调用日志、Token 与成本统计 |

> 关键设计原则：**配置随知识库走**。用户切换知识库即切换一整套 RAG 链路配置，平台负责隔离与编排。

### 4.3 平台级能力（跨用户 / 跨知识库）

- 用户与权限管理（注册、登录、角色、配额）。
- **企业身份对接**：支持对接企业身份体系标准，复用企业统一账号与组织架构，避免重复建号。支持的身份提供方包括：
  - **SSO / OAuth2**：标准单点登录与 OAuth2 授权（如 OIDC），对接企业统一认证门户。
  - **LDAP / AD**：对接企业目录服务（Active Directory / OpenLDAP），同步组织与账号。
  - **企业 IM 生态**：企业微信、钉钉、飞书，支持扫码 / 授权登录并同步部门与成员。
  - **组织架构同步**：部门 / 岗位 / 成员自动同步，并映射为平台内的用户与角色关系。
  - **接入优先级**：**SSO / OAuth2 优先接入**（对接企业统一认证门户，覆盖最广、最通用）；LDAP / AD、企业微信、钉钉、飞书按企业实际需求与排期依次扩展。所有提供方均为可插拔实现，不阻塞 MVP。
- **权限与治理**：基于部门 / 角色的访问控制、知识库可见范围、数据分级与脱敏。
- **安全合规**：操作审计日志、问答留痕、内容安全过滤、满足企业合规与等保要求。
- 全局模型 / 向量库资源池（供各知识库选用，统一接入与配额 / 计费）。
- 租户级隔离（用户 / 部门间数据、向量、日志互不越权）。
- 平台级监控与运营（活跃用户、调用量、成本大盘、知识库健康度）。

### 4.4 用户、角色与初始数据（默认内置）

采用**简单的"用户—角色"账户机制**（RBAC 雏形），初始数据可直接写入用户表与角色表：

- **默认用户**：`admin` / 密码 `123456`，初始化写入用户表（首次登录建议强制修改）。
- **默认角色**（初始化写入角色表）：
  - `系统管理员`：拥有系统管理页面，负责平台级治理。
  - `知识管理员`：拥有 RAG 管理页面，负责知识库与检索链路运营。
- **用户—角色关系**：通过用户角色关联表绑定，一个用户可拥有多个角色（MVP 可先单角色）。

**系统管理员（专属系统管理页面）能力**：
- 用户管理：新增 / 停用 / 编辑用户。
- 角色管理：新增 / 编辑角色，定义角色可见菜单与权限。
- 用户角色关系：为用户分配 / 收回角色。
- 资源配额：**按用户**分配积分配额（详见 4.6），并可管理额外积分申请的审批。
- 大模型 API 管理：配置全局可用的 LLM / Embedding / Rerank 的 API 密钥与端点，供各知识库选用。

**知识管理员（RAG 管理页面）能力**：
- 文件上传：上传并管理知识库文档。
- 数据源连接：配置本地文件、数据库、Wiki 等数据源连接器。
- 向量数据库管理：选择 / 配置知识库使用的向量库实例。
- LLM 管理：为知识库选择 / 配置生成与检索所用的大模型。

> 说明：默认采用简单本地账户机制（admin + 两角色）即可跑通 MVP；企业身份体系对接（SSO / OAuth2 / LDAP / 企业微信 / 钉钉 / 飞书，见 4.3）作为**明确支持的、可插拔接入**能力，不阻塞 MVP，但属于标准范围。接入后企业账号可自动同步并映射为平台用户与角色。

### 4.5 系统边界与对外集成（独立微服务 / 可接入 IDaaS）

- **独立微服务形态**：qoorag 作为"企业智能化"工程中一个**独立、自治的微服务**存在，不绑定其他子系统，可单独部署、扩容与演进。
- **标准 API 对外暴露**：通过 RESTful / OpenAI 兼容 API 对外提供检索与问答能力，供统一门户、智能体平台、业务系统按需调用。
- **身份可接入其他 IDaaS 服务**：在 4.3 企业身份对接基础上，支持接入企业既有 **IDaaS（身份即服务）**，复用统一身份认证，不重复建设账号体系。
- **与周边子系统的边界**：
  - **统一门户**：门户负责入口与单点登录，qoorag 通过 IDaaS / SSO 接入，不直接耦合门户。
  - **智能体平台**：可把 qoorag 当作"知识检索工具 / 技能"，通过 API 调用其检索与问答能力。
  - **数据中台**：qoorag 专注知识入库与检索，原始数据治理归属数据中台，必要时通过数据源连接器拉取。
- 设计原则：**能力内聚、接口外放、身份外置**，保持松耦合、可替换。

### 4.6 配额与计费（按用户积分制）

配额与计费的粒度确定为**按用户**，采用**积分（Credits）**机制：

- **计费主体：用户**。配额随用户走，不按部门 / 知识库单独计量（一个用户名下多个知识库共享其积分池）。
- **每月固定积分**：每个用户每月获得一份固定积分额度，周期性重置（如自然月）。
- **积分消耗**：将消耗资源的操作折算为积分扣减，主要包括：
  - 文档解析 / 向量化（Embedding 调用量）。
  - 检索与生成（LLM / Rerank 调用量、Token 消耗）。
  - （可选）存储占用等。
- **用完需额外申请**：积分耗尽后相关操作受限，用户可发起**额外积分申请**，由系统管理员审批后追加。
- **管理与可视**：
  - 系统管理员：设置每月默认积分、审批额外申请、查看各用户消耗。
  - 普通用户：查看本月剩余积分、消耗明细，并可提交额外积分申请。
- **数据模型（初步）**：用户积分账户表（当月额度 / 已用 / 剩余 / 重置周期）、积分流水表（每次扣减 / 充值记录）、额外积分申请表（申请 / 审批状态）。

> 说明：MVP 可先实现"每月固定积分 + 消耗扣减 + 耗尽拦截 + 人工审批追加"，精细化的分项折算规则后续迭代。

## 5. 核心功能构想（发散）

### 5.1 数据接入层
- 多格式文档解析：PDF、Word、Markdown、HTML、TXT、PPT、Excel。
- 数据源连接器：本地文件、URL 抓取、Notion、Confluence、Git 仓库、数据库。
- 增量同步与定时更新。
- OCR 支持（扫描件、图片中的文字）。

### 5.2 处理与索引层
- 智能分块（Chunking）：固定长度、语义分块、按标题层级分块。
- 多种 Embedding 模型支持（OpenAI、BGE、M3E、本地模型等，可插拔）。
- 向量数据库可选：Milvus、Qdrant、pgvector、Chroma、Faiss。
- 元数据管理与过滤（来源、时间、权限标签）。

### 5.3 检索层
- 向量检索 + 关键词检索（BM25）混合检索。
- 重排（Rerank）模型接入。
- 多路召回、结果去重与合并。
- 引用溯源（答案标注来源片段）。

### 5.4 生成层
- 多 LLM 支持：OpenAI、Claude、通义、DeepSeek、本地 Ollama 等，统一抽象接口。
- Prompt 模板管理与可视化调试。
- 流式输出、多轮对话上下文。
- 幻觉抑制策略（无答案时明确告知）。

### 5.5 应用与交互层
- 开箱即用的 Web 管理后台（知识库管理、上传、调试）。
- 内置对话 UI。
- 完整的 RESTful / OpenAI 兼容 API。
- SDK（Python / JS）。

### 5.6 运维与可观测
- 检索质量评估工具（召回率、命中率）。
- 调用日志、Token 统计、成本分析。
- Docker / K8s 一键部署。

## 6. 差异化亮点（怎么和别人不一样）

- **真·开箱即用**：一条命令跑通，默认配置即可用。
- **全链路可插拔**：Embedding、向量库、LLM、Rerank 每一环都能替换。
- **企业级可控 / 合规**：支持全本地部署、数据不出内网；可对接 SSO / LDAP、审计留痕、数据分级，满足企业安全与等保要求。
- **多用户多知识库隔离**：平台 / 产品形态，部门与个人自助建库，配置随知识库走。
- **轻量但可扩展**：从单机 demo 平滑升级到生产集群。
- **可观测的 RAG**：内置检索评估与调试能力，不做"黑盒魔法"。

## 7. 技术选型（已确认方向）

> 选型原则：兼顾**易用性、性能**与**企业私有化部署约束**（内网可跑、运维简单、团队易招）。

- **后端：Java + Spring Boot**。企业级主流栈，利于私有化部署、长期运维与团队招聘；通过 HTTP 客户端调用 LLM / Embedding / Rerank 等模型服务（兼容 OpenAI 等 API 规范），AI 能力以"外部服务调用"而非"内置 Python"方式集成。
- **前端：纯 HTML5 / CSS3 / JavaScript**（不依赖 React / Vue 等重型框架）。降低构建复杂度与依赖体积，便于内网分发、静态托管与长期维护。
- **数据库：PostgreSQL + pgvector**（统一存储，已确认）。元数据（用户 / 角色 / 知识库 / 配置 / 审计等）与向量 embeddings **同库存储**，部署与运维统一、私有化简单；向量检索能力由 pgvector 扩展提供。其他向量库（Milvus / Qdrant / Elasticsearch）作为可扩展选项保留。
- **任务队列 / 异步**：Spring 异步 + Redis / RabbitMQ（处理文档解析、向量化、索引等耗时任务）。
- **部署**：Docker Compose 起步（Spring Boot 服务 + 前端静态资源 + PostgreSQL(pgvector) + Redis），后续 Helm Chart / K8s。

## 8. MVP 范围建议（第一个可用版本）

聚焦"最短可用闭环"，并体现平台 / 多用户 / 多知识库形态：

1. 使用内置 `admin` 登录系统管理页面；系统管理员创建用户、角色并分配（如知识管理员）。
2. 知识管理员登录 RAG 管理页面，创建知识库（命名 + 选择一套默认六层配置）。
3. 在知识库内上传文档（PDF / Markdown / TXT）→ 自动解析 → 分块 → 向量化 → 入库。
4. 在知识库内进行问答（向量检索 + LLM 生成 + 来源引用）。
5. 一个简单的 Web 界面完成"admin 建用户/角色 → 知识管理员建库 → 上传 → 问答"全流程。
6. 私有化 Docker Compose 一键启动，满足企业内网部署。

**先不做**：多数据源连接器、复杂权限 / 角色体系、集群、评估平台、计费体系等（但需保留用户间数据隔离、基础审计日志、企业身份对接的扩展位）。

## 9. 潜在风险与挑战

- 文档解析质量参差（尤其是复杂 PDF / 表格）。
- 检索效果依赖 Embedding 与分块策略，需要大量调优。
- 多模型 / 多向量库抽象可能带来复杂度，需平衡灵活性与简洁性。
- 开源社区运营与文档质量决定项目能否被采用。
- 与 Dify、RAGFlow、FastGPT 等成熟项目的竞争与差异化。
