## 1. 一句话愿景

打造一个**企业级、偏平台 / 产品形态、可自托管、模块化可扩展**的开源 RAG（检索增强生成）系统：支持多用户管理，每位用户可管理多个相互隔离的知识库，每个知识库都拥有独立且可插拔的完整 RAG 链路，让企业能以可控、合规、低成本的方式把内部私有知识接入大模型，作为"企业智能化"工程的核心知识底座。

### 1.1 项目背景

- 本项目为企业级项目，是**企业内部"企业智能化"工程的一部分**，而非独立 ToC 产品。
- 目标是从"单点 AI 能力"走向"企业级知识智能化基础设施"：统一承载各业务线、各部门的私有知识问答与智能应用。
- 作为工程的一部分，需与既有企业 IT 体系对接（统一身份、组织架构、安全合规、运维监控），而非孤立存在。
- 强调**可控、可审计、可私有化部署**，满足企业数据安全与合规要求。

---

## 2. 我们想解决什么问题

- 大模型缺乏私有 / 实时 / 领域知识，容易"一本正经地胡说"。
- 企业内部知识分散在 Wiki、文档、工单、代码库、邮件中，员工查找与复用效率低，知识随人员流动流失。
- 现有 RAG 方案要么太重（企业级、难部署、绑定商业授权），要么太轻（demo 级、不可用于生产），缺少一个能接入企业 IT 体系、可私有化、可合规审计的开源平台。
- 各部门重复造轮子搭建问答机器人，缺乏统一的知识底座与治理能力。
- 搭建 RAG 需要拼接：文档解析、切分、向量化、检索、重排、生成……链路长、门槛高，企业难以独立运维。
- 缺乏一个"约定优于配置"、能快速跑通全流程、又能逐步深入定制、并满足企业安全合规要求的项目。

## 4. 产品形态与核心架构

### 4.1 总体形态：多用户 × 多知识库

项目定位为**平台 / 产品**形态，核心组织关系如下：

```
平台
└── 用户（账号）
    └── 知识库 A（独立、隔离）
    │   ├── 数据接入层
    │   ├── 处理与索引层
    │   ├── 检索层
    │   ├── 生成层
    │   ├── 应用与交互层
    │   └── 运维与可观测
    └── 知识库 B（独立、隔离）
        └── ……（同上六层，配置可完全不同）
```

- **多用户管理**：平台支持注册 / 登录的多个用户，用户之间数据隔离。
- **每个用户管理多个知识库**：用户可创建并管理多个知识库（如"产品文档库""内部 Wiki""客服 FAQ"），各自独立。
- **每个知识库拥有独立的六层能力**：知识库之间互不影响，且每个知识库可在六层上独立配置（例如 A 库用 BGE 向量 + pgvector，B 库用 OpenAI 向量 + Milvus；A 库用 DeepSeek 生成，B 库用本地 Ollama）。

### 4.2 知识库的独立六层（每个知识库可独立配置）

| 层级 | 知识库可独立配置的内容 |
| --- | --- |
| 数据接入层 | 接入的数据源、文档格式、同步策略、OCR 开关 |
| 处理与索引层 | 分块策略、Embedding 模型、向量库类型、元数据过滤 |
| 检索层 | 检索方式（向量 / 关键词 / 混合）、Rerank 模型、召回参数 |
| 生成层 | LLM 选型、Prompt 模板、对话策略、幻觉抑制 |
| 应用与交互层 | 对话 UI 配置、对外 API / 密钥、SDK 接入方式 |
| 运维与可观测 | 检索质量评估、调用日志、Token 与成本统计 |

> 关键设计原则：**配置随知识库走，但资源池集中管控**。知识管理员为知识库选择并编排一整套 RAG 链路配置（切换知识库即切换配置），但**模型与向量库实例均来自"系统管理员"统一接入的资源池，知识管理员不得自行接入外部模型 / 外部向量库**（见 4.7）。平台负责隔离与编排。

**知识库检索权限与共享模型（由知识管理员控制）**：
- **创建与管理归属**：知识库的创建、配置与日常运营由"知识管理员"（RAG 管理页面）负责；普通内部员工默认不拥有知识库的创建 / 管理权限。知识管理员**仅负责创建并配置知识库本身**，其下所用的模型 / 向量库实例由"系统管理员"统一接入与供给（见 4.7），知识管理员不接入外部资源。
- **检索权限由知识管理员授权**：某知识库"谁能检索 / 使用"由该知识库的知识管理员显式控制，采用 **RBAC 授权模型**：
  - **授权对象**：
    - **内部企业员工**：按**角色**授权（如某角色可检索），亦可细到部门 / 具体用户。
    - **外部用户**：经认证的外部协作方账号（如企业外部合作伙伴、临时访客凭证），其可访问的知识库由知识管理员显式授权。
  - **权限粒度**：检索 / 问答（只读使用）为主；知识库的**管理权限仍归知识管理员**，不因被授权检索而转移。
- **默认私有**：新建知识库默认仅创建者 / 同知识管理员角色可见可用（私有），需显式授权后方可被其他角色、员工或外部用户检索。
- **与数据分级联动**（见 4.8）：高密级知识库的检索授权受数据分级约束，仅授权且具备相应密级权限的用户 / 角色可检索。
- **审计**：检索权限的授予 / 收回全程留痕（见 4.8 审计），满足企业可控、可审计要求。

### 4.3 平台级能力（跨用户 / 跨知识库）

- 用户与权限管理（登录、角色、配额）。**企业内部项目不开放公网自助注册**，用户仅由系统管理员创建或经企业身份体系（SSO / LDAP 等）同步自动开通（详见 4.4）。
- **企业身份对接**：支持对接企业身份体系标准，复用企业统一账号与组织架构，避免重复建号（**替代自助注册**：企业用户通过 SSO 首次登录时按策略自动开通账号）。支持的身份提供方包括：
  - **SSO / OAuth2**：标准单点登录与 OAuth2 授权（如 OIDC），对接企业统一认证门户。
  - **LDAP / AD**：对接企业目录服务（Active Directory / OpenLDAP），同步组织与账号。
  - **企业 IM 生态**：企业微信、钉钉、飞书，支持扫码 / 授权登录并同步部门与成员。
  - **组织架构同步**：部门 / 岗位 / 成员自动同步，并映射为平台内的用户与角色关系。
  - **接入优先级**：**SSO / OAuth2 优先接入**（对接企业统一认证门户，覆盖最广、最通用）；LDAP / AD、企业微信、钉钉、飞书按企业实际需求与排期依次扩展。所有提供方均为可插拔实现，不阻塞 MVP。
- **权限与治理**：基于部门 / 角色的访问控制、知识库可见范围、数据分级与脱敏。
- **安全合规**：操作审计日志、问答留痕、内容安全过滤、满足企业合规与等保要求。
- 全局模型 / 向量库资源池（由系统管理员统一接入与纳管，供知识管理员选用，见 4.7）。
- 租户级隔离（用户 / 部门间数据、向量、日志互不越权）。
- 平台级监控与运营（活跃用户、调用量、成本大盘、知识库健康度）。

### 4.4 用户、角色与初始数据（默认内置）

采用**简单的"用户—角色"账户机制**（RBAC 雏形），初始数据可直接写入用户表与角色表：

- **默认用户**：`admin` / 密码 `123456`，初始化写入用户表（首次登录建议强制修改）。
- **默认角色**（初始化写入角色表）：
  - `系统管理员`：拥有系统管理页面，负责平台级治理。
  - `知识管理员`：拥有 RAG 管理页面，负责知识库与检索链路运营。
- **用户—角色关系**：通过用户角色关联表绑定，一个用户可拥有多个角色（MVP 可先单角色）。

**用户开通方式（企业内部项目，不开放自助注册）**：
- **系统管理员创建**：由系统管理员在系统管理页面新增用户并分配角色（MVP 默认方式）。
- **企业身份同步自动开通**：接入 SSO / LDAP 等企业身份体系后，企业用户首次通过 SSO 登录时按策略自动创建账号并映射角色（见 4.3），无需管理员逐个新增。
- **不提供公网自助注册**：无对外注册入口，账号来源仅限上述两种，符合企业内部可控、可审计的要求。

**系统管理员（专属系统管理页面）能力**：
- 用户管理：新增 / 停用 / 编辑用户。
- 角色管理：新增 / 编辑角色，定义角色可见菜单与权限。
- 用户角色关系：为用户分配 / 收回角色。
- 资源配额：**按用户**分配积分配额（详见 4.6），并可管理额外积分申请的审批。
- 大模型 API 管理：配置全局可用的 LLM / Embedding / Rerank 的 API 密钥与端点，供各知识库选用。

**知识管理员（RAG 管理页面）能力**：
- 文件上传：上传并管理知识库文档。
- 数据源连接：配置本地文件、数据库、Wiki 等数据源连接器。
- 向量数据库管理：选择 / 配置知识库使用的向量库实例。
- LLM 管理：从"系统管理员"统一接入的资源池中为知识库**选择**生成与检索所用的大模型（**不自行接入外部模型 / 外部向量库**）。
- **检索权限管理**：控制本知识库的检索 / 问答授权对象（见 4.2 检索权限与共享模型），可按角色、部门、具体内部员工，或经认证的外部用户授权 / 收回检索权限；知识库默认私有，管理权不随检索授权转移。

> 说明：默认采用简单本地账户机制（admin + 两角色）即可跑通 MVP；企业身份体系对接（SSO / OAuth2 / LDAP / 企业微信 / 钉钉 / 飞书，见 4.3）作为**明确支持的、可插拔接入**能力，不阻塞 MVP，但属于标准范围。接入后企业账号可自动同步并映射为平台用户与角色。

### 4.5 系统边界与对外集成（独立微服务 / 可接入 IDaaS）

- **独立微服务形态**：qoorag 作为"企业智能化"工程中一个**独立、自治的微服务**存在，不绑定其他子系统，可单独部署、扩容与演进。
- **标准 API 对外暴露**：通过 RESTful / OpenAI 兼容 API 对外提供检索与问答能力，供统一门户、智能体平台、业务系统按需调用。
- **身份可接入其他 IDaaS 服务**：在 4.3 企业身份对接基础上，支持接入企业既有 **IDaaS（身份即服务）**，复用统一身份认证，不重复建设账号体系。
- **与周边子系统的边界**：
  - **统一门户**：门户负责入口与单点登录，qoorag 通过 IDaaS / SSO 接入，不直接耦合门户。
  - **智能体平台**：可把 qoorag 当作"知识检索工具 / 技能"，通过 API 调用其检索与问答能力。
  - **数据中台**：qoorag 专注知识入库与检索，原始数据治理归属数据中台，必要时通过数据源连接器拉取。
- 设计原则：**能力内聚、接口外放、身份外置**，保持松耦合、可替换。

### 4.6 配额与计费（按用户：固定模型积分 + 固定向量库容量）

配额与计费的粒度确定为**按用户**，采用**"每月固定额度 + 超额申请"**机制。每个用户每月从平台获得两类固定资源额度：

- **模型积分（固定每月）**：每个用户每月获得一份固定模型积分额度，周期性重置（如自然月）。用于抵扣平台提供的模型调用，主要包括：文档解析 / 向量化（Embedding 调用量）、检索与生成（LLM / Rerank 调用量、Token 消耗）。
- **向量库容量（固定大小）**：每个用户每月 / 每周期获得固定的向量库存储容量（如固定 GB 数），其名下所有知识库共享该容量池。
- **计费主体：用户**：额度随用户走，不按部门 / 知识库单独计量（一个用户名下多个知识库共享其积分池与容量池）。
- **用完需额外申请**：任一类额度（积分或容量）耗尽后相关操作受限，用户可发起**额外申请**（额外积分 / 额外容量），由系统管理员审批后追加。
- **资源由系统管理员统一供给**：模型 / 向量库资源统一由系统管理员接入并管理（见 4.7），知识管理员仅从资源池中选用，不自行配置外部资源。
- **管理与可视**：
  - 系统管理员：设置每月默认积分 / 容量、审批额外申请、查看各用户消耗。
  - 普通用户：查看本月剩余积分 / 容量、消耗明细，并可提交额外申请。
- **数据模型（初步）**：用户资源账户表（当月积分额度 / 已用 / 剩余、向量库容量 / 已用 / 剩余、重置周期）、资源流水表（每次扣减 / 充值记录）、额外资源申请表（申请 / 审批状态）。

> 说明：本配额 / 计费机制为**产品化 / 云化能力**的设计（面向多租户 SaaS 或对外服务场景，需按用户计量与计费）。**企业内部私有化项目 MVP 不实现积分 / 容量拦截与计费**——内部部署通常统一资源、无需按用户计量；该机制作为后续演进项，待产品化（对外服务 / 云化）时再落地，精细化的分项折算规则也一并后置。

### 4.7 平台资源池与知识库独立配置的复用 / 隔离平衡

针对"平台级资源池（模型、向量库）与知识库独立配置之间如何平衡复用与隔离"，采用**两层资源模型**：模型与向量库由系统管理员统一接入与管控，知识库在资源池内自由编排、互不干扰。

1. **平台资源池（系统管理员统一接入与管控，共享基础设施）**
   - 由**系统管理员**统一接入并运维模型（LLM / Embedding / Rerank）与向量库（pgvector 等）基础设施，避免每个知识库重复建设 —— **复用**。
   - 资源池对全体知识管理员可见可选；**不允许知识管理员 / 普通用户自行接入外部模型或外部向量库**，所有模型与向量库必须经系统管理员统一纳管，以满足企业数据不出域、统一安全与合规管控（见 4.8）。
   - （产品化 / 云化场景可后续扩展按用户分配固定额度，见 4.6；企业内部 MVP 不做配额拦截。）

2. **知识库独立配置（资源池内自由编排）**
   - 知识管理员在系统管理员供给的资源池内，为不同知识库选择模型与向量库实例（配置随知识库走，见 4.2），各知识库配置互不干扰 —— **隔离**。
   - 平台负责隔离与编排：资源在系统管理员层统一纳管，配置在知识库层独立生效。

> 平衡原则：**基础设施平台化（复用）、资源系统管理员统一纳管（安全合规）、配置随知识库走（灵活隔离）**。知识管理员只创建并编排知识库，不接触底层资源接入。

### 4.8 合规与安全（企业级标准）

作为企业级项目，需满足企业通用合规要求与网络安全等级保护（等保 2.0）基本要求，对审计、留痕、数据分级提出具体约束：

**1. 审计（Audit）**
- **操作审计日志**：覆盖用户登录、角色 / 权限变更、知识库增删改、数据源配置、模型 / 向量库配置、配额调整、额外资源申请审批等关键操作，记录操作人、时间、对象、前后值。
- **全链路可追溯**：检索与生成过程（所用模型、召回片段、Prompt、输出、参数）可回溯，便于问题定位与责任认定。
- **防篡改留存**：审计日志只读、不可删除 / 篡改，定期归档，满足合规取证要求。

**2. 留痕（Traceability）**
- **问答留痕**：用户问答记录（问题、召回来源、生成答案、所用模型与参数、时间戳、用户身份）完整留存，支持复盘与合规审查。
- **知识库操作留痕**：文档上传 / 解析 / 向量化、索引变更、配置变更全程留痕。
- **日志留存周期**：满足等保 / 行业要求（一般不少于 6 个月），到期按策略归档或安全销毁。

**3. 数据分级（Data Classification）**
- **分类分级**：知识库 / 文档按敏感级别（如公开 / 内部 / 机密 / 绝密，或企业自定义分级）标注，分级策略可配置。
- **分级访问控制**：基于角色 / 部门与数据分级的访问控制，高密级知识库仅授权用户可见可用。
- **敏感数据脱敏**：入库前 / 检索后对身份证号、手机号、银行卡等 PII 自动识别与脱敏（掩码 / 替换）。
- **加密**：传输加密（TLS）、存储加密（敏感字段 / 向量库加密），密钥由企业 KMS 统一管理。

**4. 其他企业级安全要求**
- **身份鉴别与访问控制**：对接企业 SSO / LDAP（见 4.3），弱口令策略、登录失败锁定、MFA 可选。
- **密码应用安全**：符合商用密码应用安全性评估（密评）要求，敏感数据加密使用合规算法。
- **内容安全**：输入输出内容安全过滤（防注入、防敏感信息泄露、合规审查）。
- **数据不出域**：支持全私有化部署，知识数据不出企业内网（见 1.1、第 7 章）；且因外部模型 / 外部向量库统一由系统管理员纳管、知识管理员不得自行接入（见 4.7），知识片段不会发往企业外资源，从架构上保障数据不出域。

> 说明：具体等保级别（二级 / 三级）与行业附加要求（金融 / 政务）在立项后由安全合规部门核定；**MVP 阶段先落地基础审计日志、问答留痕、数据分级字段**；脱敏、密评等高级合规项后置，等保专项测评 / 整改材料后续补齐。（注：外部模型 / 向量库**不在范围内**——统一由系统管理员纳管，知识管理员不得自行接入，见 4.7。）

### 4.9 多租户隔离实现（单库 + tenant_id + PostgreSQL RLS）

为满足"用户 / 部门间数据、向量、日志互不越权"（见 4.3），采用**单 PostgreSQL 库内行级隔离**方案，兼顾部署简单与等保越权防护要求：

- **单库部署**：所有租户 / 用户的数据共存于同一个 PostgreSQL 数据库与 pgvector 扩展，不按租户分库或分 schema，降低运维复杂度（符合企业内部私有化、统一资源的特点）。
- **tenant_id 行级标记**：核心业务表（用户、知识库、文档、向量、对话、审计日志等）统一携带 `tenant_id`（企业内对应部门 / 组织单元，或个人归属），所有数据访问经应用层强制带上 `tenant_id` 过滤。
- **PostgreSQL RLS（行级安全策略）**：在数据库层启用 Row Level Security，为各表定义策略函数，使"当前会话仅能访问其 `tenant_id` 对应的行"，形成**应用层 + 数据库层双重隔离**，即使应用层疏漏也能在数据库层兜底防越权。
- **向量数据隔离**：向量表同样带 `tenant_id`，检索时 RLS 策略随查询自动附加租户条件，保证不同租户 / 用户的向量不可互查。
- **审计日志独立留存**：审计 / 操作日志表虽也带 `tenant_id`，但其保留与查询受独立权限管控（仅系统管理员 / 安全合规角色可见），删除知识库 / 用户时不连带删除审计（见 4.8 防篡改留存、第 9 章生命周期）。
- **与数据分级联动**（见 4.8）：在 `tenant_id` 隔离基础上，叠加数据分级字段与角色授权，实现"租户隔离 + 密级管控"的纵深防御。

> 说明：MVP 阶段先落地 `tenant_id` 应用层过滤 + 基础 RLS 策略；跨租户的数据共享 / 授权（如部门 KB 跨租户只读）依赖 4.2 的 RBAC 检索授权，通过显式授权记录打通，不在 RLS 层面放开。

### 4.10 对外 API 鉴权与限流

平台通过标准 API 对外提供检索 / 问答能力（见 4.5），其鉴权与限流采用如下方案：

- **按知识库签发 API Key**：每个知识库由知识管理员（或系统管理员）签发独立的 API Key，调用方凭 Key 访问**对应知识库**的检索 / 问答接口（OpenAI 兼容风格）。Key 与知识库绑定，天然继承 4.2 的检索权限与 4.9 的 `tenant_id` 隔离。
- **Key 维度鉴权**：API Key 校验通过后方可访问其绑定知识库；Key 失效 / 吊销即切断访问，无需改动用户账号体系。
- **按 Key 限流（非配额计费）**：因企业内部 MVP 不做配额 / 计费（见 4.6），限流不挂钩积分，改为**按 API Key 维度的速率限制**（如每分钟 / 每小时请求数与 Token 数上限），防止单 Key 滥用影响平台稳定性，限流策略由系统管理员配置。
- **审计关联**：每次 API 调用记录所用 Key、对应知识库、调用方、时间、用量，纳入 4.8 的审计与问答留痕。
- **安全**：Key 仅展示一次、加密存储；支持轮换与吊销；传输全程 TLS。

> 说明：MVP 先实现"按知识库签发 Key + 基础按 Key 限流 + 调用审计"；更细的按用户 / 按应用维度的 Key 管理与分级限流后续迭代。

### 4.11 知识库生命周期与数据清理

知识库从创建到删除的全过程管理，遵循"业务数据可清理、合规留存不可删"原则：

- **删除时同步清理业务数据**：删除知识库时，平台同步清理该库的全部文档、分块、元数据与向量数据（含 pgvector 中的向量行），释放存储，避免孤儿数据累积。
- **审计与留痕独立留存（不随删库删除）**：该知识库的**操作审计日志**与**问答留痕**独立于知识库业务数据表，删除知识库**不连带删除**审计 / 留痕记录，满足等保取证与责任追溯要求（见 4.8、4.9）。
- **软删除 + 保留期机制**：删除先进入"软删除（逻辑删除）"状态，标记 `deleted_at` 并暂停检索；在系统配置的**保留期**（如 30 天）内可恢复，过期后由定时任务执行物理清理（业务数据彻底删除，审计 / 留痕仍保留）。
- **级联与隔离**：清理过程受 `tenant_id` 约束（见 4.9），仅清理目标知识库所属租户数据，不影响其他租户 / 知识库；向量清理同样带 `tenant_id` 条件。
- **操作留痕**：知识库的删除 / 恢复 / 物理清理均记入审计日志（操作人、时间、对象、前后值），全程可审计。

> 说明：MVP 先实现"软删除 + 保留期 + 业务数据清理 + 审计独立留存"；跨知识库的数据归档 / 导出等高级生命周期能力后续迭代。

## 6. 差异化亮点（怎么和别人不一样）

- **真·开箱即用**：一条命令跑通，默认配置即可用。
- **全链路可插拔**：Embedding、向量库、LLM、Rerank 每一环都能替换。
- **企业级可控 / 合规**：支持全本地部署、数据不出内网；可对接 SSO / LDAP、审计留痕、数据分级，满足企业安全与等保要求。
- **多用户多知识库隔离**：平台 / 产品形态，部门与个人自助建库，配置随知识库走。
- **轻量但可扩展**：从单机 demo 平滑升级到生产集群。
- **可观测的 RAG**：内置检索评估与调试能力，不做"黑盒魔法"。

## 7. 技术选型（已确认方向）

> 选型原则：兼顾**易用性、性能**与**企业私有化部署约束**（内网可跑、运维简单、团队易招）。

- **后端：Java + Spring Boot**。企业级主流栈，利于私有化部署、长期运维与团队招聘；通过 HTTP 客户端调用 LLM / Embedding / Rerank 等模型服务（兼容 OpenAI 等 API 规范），AI 能力以"外部服务调用"而非"内置 Python"方式集成。
- **前端：纯 HTML5 / CSS3 / JavaScript**（不依赖 React / Vue 等重型框架）。降低构建复杂度与依赖体积，便于内网分发、静态托管与长期维护。
- **数据库：PostgreSQL + pgvector**（统一存储，已确认）。元数据（用户 / 角色 / 知识库 / 配置 / 审计等）与向量 embeddings **同库存储**，部署与运维统一、私有化简单；向量检索能力由 pgvector 扩展提供。其他向量库（Milvus / Qdrant / Elasticsearch）作为可扩展选项保留。
- **任务队列 / 异步**：Spring 异步 + Redis / RabbitMQ（处理文档解析、向量化、索引等耗时任务）。
- **部署**：Docker Compose 起步（Spring Boot 服务 + 前端静态资源 + PostgreSQL(pgvector) + Redis），后续 Helm Chart / K8s。

## 8. MVP 范围建议（第一个可用版本）

聚焦"最短可用闭环"，并体现平台 / 多用户 / 多知识库形态：

1. 使用内置 `admin` 登录系统管理页面；系统管理员创建用户、角色并分配（如知识管理员）。
2. 知识管理员登录 RAG 管理页面，创建知识库（命名 + 选择一套默认六层配置）。
3. 在知识库内上传文档（PDF / Markdown / TXT）→ 自动解析 → 分块 → 向量化 → 入库。
4. 在知识库内进行问答（向量检索 + LLM 生成 + 来源引用）。
5. 一个简单的 Web 界面完成"admin 建用户/角色 → 知识管理员建库 → 上传 → 问答"全流程。
6. 私有化 Docker Compose 一键启动，满足企业内网部署。

**先不做**：配额 / 计费拦截（固定积分 + 容量 + 耗尽拦截，属产品化 / 云化能力，企业内部 MVP 不实现）、多数据源连接器、复杂权限 / 角色体系、集群、评估平台、脱敏 / 密评等高级合规项（但需保留用户间数据隔离、基础审计日志、问答留痕、数据分级字段、企业身份对接的扩展位）。**外部模型 / 外部向量库接入不在范围内**——模型与向量库统一由系统管理员纳管，知识管理员仅从资源池选用（见 4.7）。

## 9. 潜在风险与挑战

- 文档解析质量参差（尤其是复杂 PDF / 表格）。
- 检索效果依赖 Embedding 与分块策略，需要大量调优。
- 多模型 / 多向量库抽象可能带来复杂度，需平衡灵活性与简洁性。
- 开源社区运营与文档质量决定项目能否被采用。
- 与 Dify、RAGFlow、FastGPT 等成熟项目的竞争与差异化。

- [x] **合规 / 等保要求**（见 4.8）：按企业级标准，满足等保 2.0 与企业通用合规，对审计（操作审计日志 + 全链路可追溯 + 防篡改）、留痕（问答留痕 + 知识库操作留痕 + 日志留存≥6 个月）、数据分级（分类分级 + 分级访问控制 + 脱敏 + 加密）提出具体约束，并涵盖身份鉴别、密评、内容安全、数据不出域。
