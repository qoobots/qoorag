## 2. 总体架构设计

### 2.1 架构风格

采用**分层单体（Modular Monolith）+ 独立微服务边界**风格：

- 当前 MVP 以单一 Spring Boot 应用承载全部能力（部署简单、私有化友好），但内部按"系统管理 / 知识库管理 / 检索问答 / 对外 API / 合规可观测"清晰分层与分包，模块间通过 Service 边界解耦。
- 保持"独立微服务"定位：未来可将检索/生成、文档处理、API 网关拆分为独立服务，当前包结构已为拆分预留边界（见 §5）。

### 2.2 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  表现层（Web / 静态前端）                                      │
│  static/：login.html index.html system-admin.html rag-admin  │
│  controller/：Auth / SystemAdmin / RagAdmin / Api             │
├─────────────────────────────────────────────────────────────┤
│  拦截与鉴权层                                                 │
│  config/AuthInterceptor：Session Token / API Key 校验         │
│  config/WebConfig：CORS、白名单（登录接口）                    │
├─────────────────────────────────────────────────────────────┤
│  业务服务层（service/）                                        │
│  Seed / Auth / User / Role / KnowledgeBase / ApiKey / Audit   │
├─────────────────────────────────────────────────────────────┤
│  数据访问层（repository/ + entity/）                           │
│  JPA Repository × 11；实体 × 11（均带 tenant_id + deleted_at） │
├─────────────────────────────────────────────────────────────┤
│  基础设施层                                                   │
│  PostgreSQL + pgvector（元数据+向量同库）                      │
│  Redis（缓存/限流，MVP 预留）/ 异步任务（文档处理）            │
│  外部模型服务（LLM / Embedding / Rerank，HTTP 调用，资源池纳管）│
└─────────────────────────────────────────────────────────────┘
```

### 2.3 包结构（已实现骨架）

```
com.qoobot.qoorag
├── QooragApplication.java        # 启动类 + BCryptPasswordEncoder Bean
├── common/                       # Result 统一响应、SessionInfo、SecurityContext(ThreadLocal)
├── config/                       # AuthInterceptor、WebConfig（VectorConverter 预留）
├── controller/                   # AuthController / SystemAdminController / RagAdminController / ApiController
├── entity/                       # 11 个 JPA 实体
├── repository/                   # 11 个 JPA Repository
└── service/                      # 7 个业务服务
```

## 3. 技术选型方案

| 层 | 选型 | 版本/说明 | 决策依据（脑暴 7） |
| --- | --- | --- | --- |
| 语言/框架 | Java 17 + Spring Boot | 3.2.5 | 企业主流栈，私有化与长期运维友好 |
| 持久化 | Spring Data JPA | — | 元数据 CRUD 标准化 |
| 关系/向量库 | PostgreSQL + pgvector | 扩展 vector(1536) | 元数据与向量同库，部署统一（4.9） |
| 前端 | 纯 HTML5 / CSS3 / JS | 无框架 | 降低构建复杂度，内网易分发（脑暴 7） |
| 密码/加密 | spring-security-crypto (BCrypt) | — | 密码与 API Key 哈希 |
| 异步/队列 | Spring 异步 + Redis / RabbitMQ | MVP 预留 | 文档解析/向量化耗时任务（脑暴 7） |
| 部署 | Docker Compose | pgvector 镜像 | 一键内网启动（MVP） |
| 外部模型 | HTTP 客户端调用（兼容 OpenAI 规范） | 资源池纳管 | AI 能力以外部服务集成，不内置 |

> 说明：MVP 骨架已引入 `spring-boot-starter-web / data-jpa / validation / postgresql / security-crypto`；pgvector Java 驱动暂以 `String`(JSON) 存向量、数据库列用 `vector(1536)`，ANN 检索待接入 pgvector JDBC 后切换（见 §10.2）。

## 4. 部署架构设计

### 4.1 MVP：Docker Compose 单机

```
[浏览器/调用方]
      │ HTTPS
      ▼
[qoorag 应用容器]  (Spring Boot :8080，静态资源内嵌)
      │ JDBC
      ▼
[postgres 容器]    (ankane/pgvector，挂载 sql/init.sql 初始化)
[redis 容器]       (MVP 预留：限流/缓存，当前 compose 未强制)
```

- `docker-compose.yml` 已定义 `postgres`（ankane/pgvector，健康探针，持久卷 `pgdata`），挂载 `sql/init.sql` 自动建表/索引/种子。
- 应用通过 `application.yml` 连接 `localhost:5432/qoorag`（容器内可改服务名）。
- 文件上传上限 50MB（multipart 配置）。

### 4.2 演进：K8s / Helm

- 后续将应用打包为镜像，PostgreSQL(pgvector) 用 StatefulSet 或托管 RDS；Redis 独立；入口 Ingress + TLS；HPA 按 CPU/调用量扩缩。

## 5. 模块/服务拆分设计

按应用架构文档（04）应用组合，内部模块边界如下（当前同进程，未来可独立部署）：

| 模块 | 主要类 | 职责 |
| --- | --- | --- |
| 系统管理 | SystemAdminController + UserService + RoleService | 用户/角色/关系、资源池入口 |
| 知识库管理 | RagAdminController + KnowledgeBaseService | KB CRUD、文档、检索权限、Key、生命周期 |
| 认证鉴权 | AuthController + AuthService + AuthInterceptor | 登录、API Key 签发/校验、会话 |
| 对外 API | ApiController + ApiKeyService | /retrieve、/chat（OpenAI 兼容） |
| 合规可观测 | AuditService | 审计日志、问答留痕写入 |
| 初始化 | SeedService | 默认租户 + 两角色、admin 账号写入 |

> 拆分原则（4.5）：能力内聚、接口外放、身份外置。检索/生成、文档处理是可独立成服务的候选边界。

## 6. 中间件架构

- **PostgreSQL + pgvector**：唯一持久化中间件，元数据与向量同库。向量列 `vector(1536)`，按 embedding 模型维度调整。
- **Redis（预留）**：API Key 限流计数、热点缓存（如权限表）、异步任务队列备选。MVP 限流可先基于内存/数据库字段，Redis 接入不阻塞。
- **异步处理**：文档解析→分块→向量化为耗时链路，采用 Spring `@Async` + 线程池（或后续 RabbitMQ），避免阻塞上传请求；状态机：PENDING→解析中→向量化中→已入库/失败。
- **外部模型服务**：LLM/Embedding/Rerank 经 HTTP 调用，端点与密钥由系统管理员在资源池配置（不暴露给知识管理员）。

---

## 7. 高可用与容灾备份

- **数据备份**：PostgreSQL 定期 `pg_dump` / 物理备份（WAL 归档）；向量随元数据同库，备份一体。
- **保留与清理**：知识库软删除保留期（`kb-retention-days: 30`，见 application.yml），过期定时任务物理清理业务数据；审计/留痕独立保留。
- **可用性（MVP）**：单机 Compose 起步；后续 K8s 多副本 + 数据库主从/托管，消除单点。
- **故障恢复**：容器重启自动重建连接；DB 持久卷保障数据不丢；种子数据幂等（ON CONFLICT DO NOTHING）。

## 8. 安全基础设施设计

### 8.1 身份与访问控制

- **登录鉴权**：`AuthController.login` 校验 BCrypt 密码，签发 Session Token（存入 SecurityContext/ThreadLocal，由拦截器注入请求上下文）。
- **API Key 鉴权**：`AuthInterceptor` 识别 `Authorization: Bearer <apiKey>`，经 `ApiKeyService` 校验 `key_hash`（SHA-256），定位绑定知识库，继承其 `tenant_id` 隔离与 RBAC 检索权限。
- **白名单**：登录接口放行，其余需鉴权；CORS 由 `WebConfig` 配置。
- **RBAC**：角色（系统管理员/知识管理员）决定可访问的 Controller/菜单；知识库检索权限由 `kb_permission` 表细粒度控制（ROLE/USER/EXTERNAL）。

### 8.2 数据安全与合规（4.8）

- **传输加密**：全链路 TLS（生产前置反向代理/Ingress）。
- **存储加密**：密码 BCrypt；API Key 仅存哈希（`key_hash`），明文仅展示一次。
- **数据不出域**：模型/向量库由系统管理员统一纳管，知识管理员不得接外部资源；外部模型调用仅限企业内网部署实例。
- **审计防篡改**：`audit_log` / `qa_trace` 只读、不随删库删除，满足等保取证。
- **数据分级**：知识库/文档带 `data_classification`，高密级仅授权且具密级权限者检索。
- **后置项**：PII 脱敏、密评、内容安全过滤细化（见 02 §7.2）。

## 9. 关键技术设计

### 9.1 认证鉴权流程

```
请求 → AuthInterceptor
  ├─ Header 含 Bearer <apiKey> → ApiKeyService 校验 key_hash → 绑定 KB + tenant_id → 放行 API 接口
  └─ Header 含 Session Token   → 解析用户/角色/tenant → 注入 SecurityContext → 放行管理接口
登录：AuthController → AuthService 校验 BCrypt → 签发 Token
```

### 9.2 RAG 处理流水线（知识库内）

```
上传文档(PDF/MD/TXT)
  → 解析（文本抽取，复杂格式后置）
  → 分块（固定长度/语义/标题层级，MVP 可配）
  → Embedding 向量化（资源池模型，HTTP 调用）
  → 写入 vector_data（tenant_id + kb_id + embedding）
问答：
  问题 → Embedding → 向量检索（pgvector 余弦/内积，带 tenant_id + kb_id 过滤）
       → [可选 Rerank] → Top-K 片段 → LLM 生成 + 来源引用 → 写入 qa_trace
```

> 向量存储当前以 `String`(JSON) 映射 `vector(1536)` 列；接入 pgvector JDBC 驱动后改为原生 `PGvector` 类型以支持 ANN 索引（IVFFlat/HNSW）加速检索。

### 9.3 多租户隔离实现

- 应用层：所有 Repository 查询显式带 `tenant_id`（由 SecurityContext 注入）。
- 数据库层：RLS 策略兜底（见 03 §5.2 启用步骤）。
- 向量隔离：`vector_data` 带 `tenant_id`，检索查询自动附加租户条件。

### 9.4 知识库生命周期（4.11）

```
删除 → 置 deleted_at（软删，暂停检索）
  → 保留期(kb-retention-days)内可恢复
  → 过期定时任务：物理删除 document/chunk/vector_data（受 tenant_id 约束）
  → audit_log / qa_trace 独立保留（不删）
```

### 9.5 对外 API（4.10，OpenAI 兼容）

- `POST /api/v1/retrieve`：检索召回片段。
- `POST /api/v1/chat`：检索+生成问答。
- 鉴权：按 KB 签发的 API Key；限流：按 Key `rate_limit`（次/周期），由系统管理员配置；调用写入审计与 qa_trace。
